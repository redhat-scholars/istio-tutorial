Advanced Route Rules

Before Start

You should have NO virtualservice nor destinationrule (in tutorial namespace) istioctl get virtualservice istioctl get destinationrule if so run:

./scripts/clean.sh tutorial-$WORKSHOP_USER

Smart routing based on user-agent header (Canary Deployment)

What is your user-agent?

https://www.whoishostingthis.com/tools/user-agent/

Note: the "user-agent" header is added to OpenTracing baggage in the Customer service. From there it is automatically propagated to all downstream services. To enable automatic baggage propagation all intermediate services have to be instrumented with OpenTracing. The baggage header for user agent has following form baggage-user-agent: <value>.

Set recommendation to all v1

istioctl create -f istiofiles/destination-rule-recommendation-v1-v2.yml -n tutorial-$WORKSHOP_USER
istioctl create -f istiofiles/virtual-service-recommendation-v1.yml -n tutorial-$WORKSHOP_USER

Set Safari users to v2

istioctl replace -f istiofiles/virtual-service-safari-recommendation-v2.yml -n tutorial-$WORKSHOP_USER

istioctl get virtualservice -n tutorial-$WORKSHOP_USER

and test with a Safari (or even Chrome on Mac since it includes Safari in the string). Safari only sees v2 responses from recommendation

and test with a Firefox browser, it should only see v1 responses from recommendation.

There are two ways to get the URL for your browser:

Open the url http://customer-tutorial-$WORKSHOP_USER.apps.testans-f011.openshiftworkshop.com in your broswer

You can also attempt to use the curl -A command to test with different user-agent strings.

curl -A Safari customer-tutorial-$WORKSHOP_USER.apps.testans-f011.openshiftworkshop.com
curl -A Firefox customer-tutorial-$WORKSHOP_USER.apps.testans-f011.openshiftworkshop.com

You can describe the virtualservice to see its configuration

istioctl get virtualservice -o yaml -n tutorial-$WORKSHOP_USER

Remove the Safari rule

istioctl delete -f istiofiles/virtual-service-safari-recommendation-v2.yml -n tutorial-$WORKSHOP_USER

Set mobile users to v2

istioctl create -f istiofiles/virtual-service-mobile-recommendation-v2.yml -n tutorial-$WORKSHOP_USER

curl -A "Mozilla/5.0 (iPhone; U; CPU iPhone OS 4(KHTML, like Gecko) Version/5.0.2 Mobile/8J2 Safari/6533.18.5" http://customer-tutorial-$WORKSHOP_USER.apps.testans-f011.openshiftworkshop.com

Clean up

istioctl delete -f istiofiles/destination-rule-recommendation-v1-v2.yml -n tutorial-$WORKSHOP_USER
istioctl delete -f istiofiles/virtual-service-mobile-recommendation-v2.yml -n tutorial-$WORKSHOP_USER

or you can run:

./scripts/clean.sh tutorial-$WORKSHOP_USER

Mirroring Traffic (Dark Launch)

oc get pods -l app=recommendation -n tutorial-$WORKSHOP_USER
or
kubectl get pods -l app=recommendation -n tutorial-$WORKSHOP_USER

You should have 2 pods for recommendation based on the steps above

istioctl get virtualservice -n tutorial-$WORKSHOP_USER
istioctl get destinationrule

You should have NO virtualservice nor destinationrule (in tutorial namespace) istioctl get virtualservice istioctl get destinationrule if so run:

./scripts/clean.sh tutorial-$WORKSHOP_USER

Make sure you are in the main directory of "istio-tutorial"

istioctl create -f istiofiles/destination-rule-recommendation-v1-v2.yml -n tutorial-$WORKSHOP_USER
istioctl create -f istiofiles/virtual-service-recommendation-v1-mirror-v2.yml -n tutorial-$WORKSHOP_USER

curl customer-tutorial-$WORKSHOP_USER.apps.testans-f011.openshiftworkshop.com

Check the logs of recommendation-v2

oc logs -f `oc get pods|grep recommendation-v2|awk '{ print $1 }'` -c recommendation
or
kubectl logs -f `oc get pods|grep recommendation-v2|awk '{ print $1 }'` -c recommendation

Clean up

istioctl delete -f istiofiles/destination-rule-recommendation-v1-v2.yml -n tutorial-$WORKSHOP_USER
istioctl delete -f istiofiles/virtual-service-recommendation-v1-mirror-v2.yml -n tutorial-$WORKSHOP_USER

or you can run:

./scripts/clean.sh tutorial-$WORKSHOP_USER

Load Balancer

By default, you will see "round-robin" style load-balancing, but you can change it up, with the RANDOM option being fairly visible to the naked eye.

Add another v2 pod to the mix

oc scale deployment recommendation-v2 --replicas=2 -n tutorial-$WORKSHOP_USER
or
kubectl scale deployment recommendation-v2 --replicas=2 -n tutorial-$WORKSHOP_USER

Wait a bit (oc get pods -w to watch) and curl the customer endpoint many times

curl customer-tutorial-$WORKSHOP_USER.apps.testans-f011.openshiftworkshop.com

Add a 3rd v2 pod to the mix

$ oc scale deployment recommendation-v2 --replicas=3 -n tutorial-$WORKSHOP_USER
$ oc get pods -n tutorial-$WORKSHOP_USER

or

$ kubectl scale deployment recommendation-v2 --replicas=3 -n tutorial-$WORKSHOP_USER
$ kubectl get pods -n tutorial-$WORKSHOP_USER
